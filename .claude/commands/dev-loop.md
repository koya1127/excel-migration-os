# 自律開発ループ

与えられた改善目標に対して、以下のサイクルを自律的に繰り返す。
人間の介入なしに、品質が合格ラインに達するまで回し続ける。

## 引数
$ARGUMENTS — 改善目標（例: 「使い方シートのメニュー項目の重複を減らす」「Python出力のmain.pyが関数を自動呼び出しするようにする」）

## ループ手順

### Step 1: 現状確認
- 改善目標に関連するコードを読む
- 必要ならPlaywright MCPまたはブラウザで本番サイトの現状を確認
- 問題点を特定して箇条書きにする

### Step 2: 修正実装
- 特定した問題に対してコードを修正する
- `dotnet build`（バックエンド）または `npx next build`（フロントエンド）でビルド確認
- ビルドエラーがあれば修正

### Step 3: コミット＆デプロイ
- `git add` → `git commit` → `git push origin main`
- GitHub Actions のデプロイ完了を待つ（`gh run list` で確認、3-4分）

### Step 4: E2Eテスト
- Playwright MCPで本番サイトにアクセス
- テスト対象の機能を実際に操作して確認
- スクリーンショットを撮って結果を記録

### Step 5: 評価
- 改善目標が達成されたか判定
- まだ問題があれば → Step 1 に戻る（次のイテレーション）
- 問題なければ → 完了報告

## ルール
- 各イテレーションでコミットメッセージに何を修正したか明記する
- 最大10イテレーションで打ち切り（無限ループ防止）
- デプロイ待ちはバックグラウンドタスクで（`sleep` + `gh run list`）
- テストは本番環境（Vercel + Azure Container Apps）で実施
- 問題が見つかったら即修正、迷ったら先に進む
- 各ステップの結果を簡潔に記録する
